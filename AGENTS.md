## Package Overview

**arsworks** is a monorepo of 6 interdependent R packages that implement CDISC ARS (Analysis Results Standards) v1.0 for clinical table generation and reporting.

**Packages (in dependency order):**
1. **arscore** — Foundation: data structures, validation, base classes
2. **arsshells** — Builder/Factory: constructs arscore objects from templates
3. **arsresult** — Executor: transpiles conditions, executes methods on data
4. **arstlf** — Translator: ARD → tfrmt/gt → publication-ready tables
5. **ars** — Orchestrator: pipe API, bundled datasets, end-user entry point
6. **arsstudio** — Shiny UI: interactive browser for the full hydrate/run/render pipeline

## CDISC & Clinical Standards

### ARS Requirements
- All tables must conform to CDISC ARS v1.0 specification
- Metadata-driven derivation (no hardcoded logic)
- Reproducible, version-controlled output

### ADaM Integration
- Input data: ADaM datasets (SDTMIG v3.4, ADaMIG v1.3)
- Use {pharmaversesdtm} for test data
- Leverage {admiral} + {admiralonco} conventions
- Define-XML v2.1 metadata via {metacore}

### Pharmaverse Ecosystem
- Upstream: {admiral}, {admiralonco}, {metacore}
- Parallel: {xportr} (transport to XPT)
- Downstream: {rtables}, {tern}, {teal} (for analysis workflows)
- Testing: {testthat} + {admiralvaccine} patterns

### Response Criteria
- **RECIST 1.1** for oncology trials (primary)
- **SDTMIG Oncology** Disease Response Supplement (v2.0)
- TU/TR/RS domains with RELREC linkage

## Design Principles

### 1. Metadata-Driven
- Never hardcode business logic
- Specifications live in {metacore} (or JSON/YAML equivalent)
- Code reads specs and derives outputs

### 2. Separation of Concerns
- **arscore**: Data models only
- **arsshells**: Template construction
- **arsresult**: Computation engine
- **arstlf**: Rendering/output

### 3. Immutability
- Derivatives never modify input data
- Return new objects; preserve lineage
- Enable audit trails for regulatory use

### 4. Type Safety
- Use S7 classes (or S4 if S7 unavailable)
- Validate on construction, not in methods
- Fail fast with clear error messages

## Code Conventions

### File Structure

Each package follows standard R-package layout. arscore's convention —
**one file per S7 class** — is the model the others largely follow:

```
arscore/
├── R/
│   ├── arscore-package.R                 # Package docs + .onLoad
│   ├── validate_helpers.R                # Shared validator helpers
│   ├── ars_<class>.R                     # One file per S7 class (38 of these)
│   ├── ars_json.R                        # JSON I/O across all classes
│   ├── validate_reporting_event.R        # Cross-reference integrity
│   ├── create_ard.R                      # ARD extraction + enrich_ard()
│   └── format_result_pattern.R           # Operation.resultPattern formatter
├── tests/testthat/test-*.R               # One test file per class (+ integration)
├── vignettes/*.Rmd                       # Long-form tutorials
├── man/                                  # Auto-generated Rd files (never edit directly)
├── tools/                                # Reproducible audit/diagram scripts
├── DESCRIPTION                           # Includes the Collate field that controls load order
├── NAMESPACE                             # Auto-generated by roxygen2
├── _pkgdown.yml                          # pkgdown site config
└── README.md
```

DESCRIPTION's `Collate:` field controls load order — necessary because
later S7 classes reference earlier ones. After adding a new class file,
run `devtools::document()` to update `Collate`, `NAMESPACE`, and `man/`.

### Naming Conventions

- **S7 classes (data model)**: snake_case with `ars_` prefix —
  `ars_analysis`, `ars_method`, `ars_reporting_event`. Constructors use
  the matching `new_ars_*()` form.
- **User-facing wrapper classes (orchestration)**: PascalCase — `Shell`
  in arsshells, `ArsResult` in arsresult/ars.
- **Functions**: snake_case (e.g. `validate_reporting_event`,
  `create_ard`, `enrich_ard`, `use_shell`).
- **Internal functions**: prefix with `.` (e.g. `.compact()`,
  `.normalize_data()`).
- **S7 properties**: snake_case in R; the JSON layer translates to
  camelCase to match the ARS schema (e.g. `method_id` ↔ `methodId`).
  Read with `obj@property`, not `@`-as-method-dispatch.
- **S7 methods**: dispatched via `method(generic, class) <- function(x, ...) { ... }`.
  Not the same as `@` (property access).

### Documentation
- Use roxygen2 `#'` comments
- Every exported function must have `@examples`
- Include `@seealso` cross-references between packages
- Run `devtools::document()` — never edit man/ files directly

### Testing
- Unit tests in `tests/testthat/test-*.R`
- Use `{testthat}` + `{expect_dfs_equal()}` for datasets
- Test typical use, edge cases, and error conditions
- Every new function requires tests before PR merge

## Data Handling

### ADaM Conventions
- Flag variables (ANL01FL, SAFFL, etc.) take "Y" or NA — never "N"
- PARAM/PARAMCD must align with CDISC Controlled Terminology
- ASEQ derived as last step before finalization
- DTYPE for derived records follows SDTM conventions

### Test Data
- Use `{pharmaversesdtm}` for CDISC Pilot 01 SDTM data
- Generate ADaM via {admiral} for arsworks tests
- Real-world example: See `data_table_examples.R`

### Output Format
- Primary: data.frame with CDISC-compliant structure
- Secondary: ARD (Analysis Results Dataset) via {pharmaverse}
- Exported via {xportr} to XPT for submission

## Git Workflow

### Branch Naming
- **Feature**: `feature/description` (e.g., `feature/recist-response-flag`)
- **Bug fix**: `fix/description` (e.g., `fix/aseq-derivation-order`)
- **Docs**: `docs/description`
- **Release**: `release/vX.Y.Z`

### Commit Messages
- Format: `type(package): short description`
- Types: `feat`, `fix`, `docs`, `test`, `refactor`, `chore`
- Example: `feat(arsresult): add condition transpiler for RECIST response`

### PRs
- Link to GitHub issue (if applicable)
- Include before/after examples
- Update CHANGELOG.md
- Ensure `devtools::check()` passes (no ERRORs or WARNINGs)

## Testing Strategy

### Unit Tests (Required)
- Every exported function
- Test typical use case
- Test edge cases (missing data, empty inputs, boundary values)
- Test error conditions with appropriate error messages

### Integration Tests
- Test cross-package workflows (e.g., arscore → arsshells → arsresult)
- Use snapshot tests for complex output validation
- Reference real clinical trial examples

### Test Execution

```r
# Local — single package
devtools::test("arscore")

# Local — whole suite via the monorepo entry point (loads all 6 packages,
# runs the example tables and §22 UAT checks)
source("setup.R")

# Pre-PR check
devtools::check("arscore", quiet = TRUE)  # 0 errors / 0 warnings / 0 notes target
```

**Suite-wide test totals** (measured 2026-04-30): 1,163 tests / 2,477
expectations / 0 failures across all 6 packages. Per-package counts and
versions live in `MASTER_PLAN.md` §0.

## Documentation Standards

### Function Documentation

Real example pattern, taken from `arscore::validate_reporting_event()`:

```r
#' Validate referential integrity of an ARS reporting event
#'
#' Checks that all ID-based cross-references resolve to objects in the
#' appropriate top-level collections, and emits non-fatal warnings for
#' defined-but-unreferenced methods, groupings, sets, and subsets.
#'
#' @param x An `ars_reporting_event` object.
#' @param strict Logical. When `TRUE`, additionally enforce schema
#'   *required* fields that the default validator tolerates (e.g.
#'   `main_list_of_contents`). Defaults to `FALSE`.
#'
#' @return Invisibly returns `TRUE` on success; otherwise raises a
#'   `cli`-formatted error listing every broken reference found.
#' @export
#'
#' @seealso [`create_ard()`], [`enrich_ard()`],
#'   [`reporting_event_to_json()`]
#'
#' @examples
#' library(arscore)
#' re <- new_ars_reporting_event(
#'   id = "RE_DEMO", name = "Demo", version = 1L,
#'   methods = list(new_ars_method(
#'     id = "METH_DESC", name = "Descriptive",
#'     operations = list(new_ars_operation(id = "OP_N", name = "N", order = 1L))
#'   ))
#' )
#' validate_reporting_event(re)
```

## Regulatory & Compliance

### CDISC Alignment
- All outputs must be SDTMIG v3.4 / ADaMIG v1.3 compliant
- Follow CDISC Controlled Terminology 2024-03
- Document any deviations from standard with rationale

### Audit Trail
- Preserve lineage: input → intermediate → output
- Include version metadata (package versions, R version, date)
- Enable reproducibility for regulatory submissions

### Code Review Checklist
- [ ] CDISC standards compliance verified
- [ ] ADaM conventions followed (flags, SEQ variables, etc.)
- [ ] Cross-package integration tested
- [ ] Documentation complete and linked
- [ ] No hardcoded logic (all metadata-driven)

## Common Mistakes to Avoid

1. **Hardcoding business logic** — Use metadata specs instead
2. **Modifying input data in place** — Always return new objects
3. **Skipping documentation examples** — Examples are executable tests
4. **Ignoring CDISC conventions** — Flag variables, ASEQ, PARAM/PARAMCD
5. **Not testing edge cases** — Missing data, empty groups, single subjects
6. **Editing man/ files directly** — Always use roxygen2 and `devtools::document()`
7. **Cross-package assumptions** — Versions can differ; test both old and new

## AI Assistant Guidance

If you are an AI assistant contributing to arsworks:

1. **Read this file plus the package's own AGENTS.md first** — they define the
   project context. Then check `MASTER_PLAN.md` §0 for current state.
2. **Understand the dependency order** —
   `arscore → arsshells / arsresult → arstlf → ars → arsstudio`.
3. **PRs go to the submodule repos**, not arsworks. See `CONTRIBUTING.md`.
   arsworks itself only takes commits that bump submodule pointers or edit
   the meta-files (bootstrap.R, setup.R, the markdown docs, renv.lock).
4. **Never hardcode clinical logic** — all specs come from metadata
   (shell templates + ARS objects).
5. **Test thoroughly** — every function, every edge case. Aim for
   `devtools::check()` at 0 errors / 0 warnings / 0 notes before PR.
6. **Document as you code** — roxygen2 comments + `devtools::document()`.
   Never edit `man/` files directly.
7. **Read the schema before changing the model** —
   `arscore/SCHEMA_CONFORMANCE.md` documents every divergence from the
   CDISC spec; new divergences should be deliberate and added there.
8. **Open the UML diagram tool** to orient on cross-class wiring before
   refactoring: `Rscript arscore/tools/uml_diagram.R` then open the
   resulting HTML.
9. **Link to CDISC specs** when relevant — reference standards, not just code.
10. **Ask for clarification** — if a spec is ambiguous, flag it in the PR
    description rather than picking an interpretation silently.
